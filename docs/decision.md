# 設計判断ログ

このドキュメントは「なぜそうしたか」「何を捨てたか」「何を見送ったか」を残す。READMEは "What it does"、これは "Why it is shaped this way"。

---

## 1. なぜ自作したか — NJSSを使わない理由

NJSSなどの有料入札情報サービスは年額数万〜十数万円の固定費。当社（メンバー1-2名・実績まだ少なめ）の規模では:

- 月数件しか入札せず単価で割ると高すぎる
- 検索条件のチューニングが粗い（業種コード単位で「データ分析」がうまく拾えない）
- 案件単位のメモ・応募履歴の蓄積ができない

→ 自社のキーワード辞書 + 自社の判断ロジックを持てる方が長期的に価値がある。**サブスク費用 = ビルド予算** と読み替えて自作。

---

## 2. なぜ二段フィルタか — ルール + LLM

```
[全案件 数百〜数千件/日]
   ↓ ルールフィルタ (キーワード辞書 + 除外語)
[2点以上 数十件/日]
   ↓ LLM判定 (Haiku、上位N件のみ)
[応募候補 数件/日]
```

検討した代替案と却下理由:

| 案 | 却下理由 |
|----|---------|
| **全件LLM判定** | コスト爆発。1案件1500トークン × 数百件 = 月数千〜数万円。スパム的な案件にトークンを払う合理性がない |
| **ルールのみ** | キーワード「データ」「調査」が含まれるが本質的に違う案件（例:「住民意識調査」と「データ可視化基盤構築」）を区別できない。文脈判断はLLMでないと無理 |
| **Embedding類似度** | 過去の応募案件が少なく学習データが揃わない。コールドスタート問題 |

ルールは粗く広く拾い、LLMで文脈判断する。**二段の役割分担が一番ROIが高い。**

---

## 3. なぜClaude Haiku 4.5か — Sonnet/Opusではない理由

応募可否判定のタスク特性:

- 入力: 公告文 1500トークン程度
- 出力: 構造化JSON（verdict / confidence / reason / effort / concerns）
- 必要な能力: テキスト読解 + 軽い推論 + JSON厳格出力

→ Haiku 4.5で十分。Sonnetは過剰。Opusは論外。

コスト試算（仮）:
- 入力 ~2000 token / 案件、出力 ~300 token
- 1日10件判定 × 30日 = 月300件
- 月コスト数百円規模 → 個人事業の固定費として許容範囲

**速度も効く**: Haikuは数秒で返るので、毎日の通知に間に合わせやすい。

## 4. なぜローカルSQLiteか

検討した代替案:

| 案 | 却下理由 |
|----|---------|
| **Postgres / Supabase** | この規模(数千レコード/年)では過剰。サーバー代がもったいない |
| **BigQuery / Cloud SQL** | Phase 1の検証段階で月課金は重い。データ量が10万件超えてから移行で間に合う |
| **Google Sheets** | 差分検知が辛い。url_hashによる重複排除がSQLの方が楽 |

SQLiteは:
- ファイル1個でバックアップ簡単
- url_hash(SHA256) で UNIQUE制約 → 差分検知が一行
- マイグレーション不要のシンプルさ

**Phase 3でクラウド移行する前提**で、Phase 1-2はローカルでスピード優先。

## 5. なぜ「個別パーサー」を書かないか

55自治体それぞれのHTMLを個別にパースする選択肢は早期に却下:

- 自治体サイトの改修頻度は高い（数年に1回はリニューアル）→ 保守地獄
- 似たCMSパターンが多い（`<table>` / `<ul><li>` / `<dl>`）

→ 汎用パーサーで4パターンをベストエフォート抽出 + raw_textをDBに残す。
→ 取りこぼしはLLM側でリカバリーする想定。

「**完璧な抽出より、80%の抽出 + LLMでの文脈補正**」の方が総合精度が高くてメンテも楽。

## 6. スコアリング設計の根拠

`filter/matcher.py` の加点ロジック:

```
high_priority   +3点  データ分析 / ダッシュボード / BI / DX等
medium_priority +2点  統計調査 / 集計 / アンケート等
low_priority    +1点  調査業務 / コンサルティング等
プロポーザル種別 +2点
終了済案件 (審査結果 等) スコア半減
```

意図:
- **高/中/低の3段階**: 直接的な領域(data/BI)と隣接領域(調査/コンサル)で重みを変える
- **プロポーザルボーナス**: 入札より企画提案型の方が小規模事業者でも勝ち筋がある
- **終了済の半減 (除外ではない)**: タイトルに「審査結果」が入っていても、たまに次の公告へのリンクが含まれることがあるので0点ではなく半減

`notify_threshold: 2` の根拠: low_priority 1個だけだと弱すぎる、最低でも mid_priority 1個 か low 2個。

## 7. やらない選択 (Phase 1で意図的に外したもの)

- **PDF解析**: 多くの公告がPDFのみ。今は title + URL のみ保存し、LLM判定時に必要なら別パスで対応予定 (Phase 2-3)
- **応募締切の自動抽出**: 公告ページからの日付抽出は構造が揺れすぎて精度出ない。raw_textに含める形で人間判断
- **応募書類の下書き生成**: 一気にやると検証コストが高い。判定エージェントが安定してから (Phase 3)
- **Web UI**: Slack通知で十分。UIに時間を使うなら判定精度に投資する

「**やらない**」を明示することで、レビューしやすく、後で「なぜやってないんですか」に答えられる。

---

## 8. 失敗・試したけど捨てたもの

- **Embeddingベースの類似度フィルタ**: 過去応募データが少なすぎてゼロショットでは精度出ず。データ蓄積が進んだら再検討
- **個別自治体のCSSセレクタ手書き**: 3自治体やった時点で保守不可能と判断、汎用パーサーに切り替え
- **`title <= 5文字` での除外**: 元々入れていなかったが、パンくずリンク（"入札"だけのテキスト）を拾って通知が荒れたので追加

## 9. ロードマップと判断軸

| Phase | 内容 | 移行判断 |
|-------|------|---------|
| 1 (完了) | 四国MVP + LLM判定 | DBに数百件貯まり、ノイズ率を計測できたら |
| 2 | 東北・九州拡大、cron常駐、PDF対応、落札結果収集 | 月の応募実績が3件超えたら |
| 3 | Cloud Functions + BigQuery、応募書類下書き生成 | 月コストが数千円超えたら、または案件が10万件超えたら |

各 Phase の終わりで **「次に行く価値があるか」を再判断** する。
